# Проблемы текущей архитектуры

## 1. Бэкенд (Go)

| Проблема | Описание |
|----------|----------|
| **Отсутствие чёткого domain-слоя** | `core/` содержит сущности и интерфейсы репозиториев, но нет доменных сервисов, агрегатов или бизнес-правил. «Домен» — это просто структуры данных. |
| **Команды смешаны с инфраструктурой** | `commands/` содержит логику use-case'ов, которая напрямую вызывает инфраструктуру (файловую систему, runtime). Нет слоя application-сервисов, который бы управлял взаимодействием между доменом и инфраструктурой. |
| **Обработчики знают слишком много** | `interfaces/wailshandler/` имеет 9 обработчиков, каждый из которых напрямую вызывает `commands.Manager`. Обработчики также занимаются локализацией ошибок, смешивая ответственности. |
| **Link preview и обработка изображений в бэкенде** | Это задачи презентационного слоя. Бэкенд должен предоставлять примитивы (чтение файла, открытие URL), а не разрешать YouTube-embed'ы или конвертировать изображения в base64. |
| **Управление хранилищами — команда бэкенда** | Создание/удаление vault'ов — это логика оркестрации, которая принадлежит фронтенду. Бэкенд должен предоставлять только файловые примитивы. |
| **Настройки — команды бэкенда** | `getLocalization`, `setLocale` — команды бэкенда. Настройки должны быть функцией фронтенда с KV-персистентностью бэкенда. |

## 2. Фронтенд (React/TypeScript)

| Проблема | Описание |
|----------|----------|
| **Нет чёткой границы ядра** | Всё находится в `widgets/`, `entities/`, `features/` без явного разделения на «ядро» и «плагины». Редактор, файловое дерево, вкладки, поиск и система плагинов имеют одинаковый вес. |
| **Функции редактора монолитны** | `editor-panel/` содержит Tiptap setup, расширения, автосохранение, обработку изображений, поиск по файлу, TOC, slash-команды, сессии плагинов — всё в одной директории без архитектурного разделения. |
| **Файловое дерево вшито в ядро** | Сайдбар с файловым деревом — это core-виджет, но он должен быть плагином (по выбору пользователя). То же самое с поиском. |
| **Реестр плагинов — хранилище сущностей** | `entities/plugin/model/pluginRegistry.ts` смешивает UI-регистрации плагинов с их жизненным циклом. Нет чёткого разделения между ядром плагинов (загрузка, песочница) и функциями плагинов (регистрации). |
| **Нет абстракции менеджера рабочих пространств** | `WorkspaceShell.tsx` — монолитный компонент. Нет явного `WorkspaceManager`, который управляет жизненным циклом рабочего пространства, раскладкой панелей, кнопками панели инструментов плагинов и т.д. |
| **Обработка изображений и link preview — вызовы бэкенда** | `imageApi.ts` и `linkPreviewApi.ts` вызывают обработчики бэкенда. Это должны быть сервисы фронтенда, использующие базовые примитивы бэкенда. |

## 3. Плагин API

| Проблема | Описание |
|----------|----------|
| **Нет хуков жизненного цикла** | Плагины выполняются через `new Function('api', source)` без хуков `onLoad`, `onUnload`, `onSettingsChange`. Очистка выполняется вручную и подвержена ошибкам. |
| **Слабые типы TypeScript** | Интерфейс `VoltPluginAPI` обширен, но не имеет дженериков, строгих типов возврата и улучшенного автодополнения IDE для разработчиков плагинов. |
| **Нет межплагинного взаимодействия** | Плагины не могут безопасно общаться друг с другом. |
| **Настройки плагинов — только автогенерация** | Плагины не могут рендерить собственный UI настроек. |

## 4. Дизайн-система

| Проблема | Описание |
|----------|----------|
| **60+ CSS-токенов без принудительного контроля** | `themeRegistry.ts` определяет токены, но компоненты могут (и делают) использовать захардкоженные цвета, отступы и значения border-radius вне системы токенов. |
| **UI-компоненты непоследовательны** | `shared/ui/` имеет 12 компонентов (button, text-input, select, toggle, modal и т.д.), но не все они следуют одинаковым паттернам отступов, border-radius, hover-состояний или фокусных колец. |
| **Нет шкалы отступов** | Нет токенов `--spacing-xs`, `--spacing-sm` и т.д. Компоненты используют произвольные значения `px` и `rem`. |
| **Нет шкалы border-radius** | Компоненты используют произвольные значения `border-radius` (4px, 6px, 8px, 12px) без систематической шкалы. |
| **Нет линт-контроля** | Нет правил ESLint или stylelint, предотвращающих захардкоженные значения. |
